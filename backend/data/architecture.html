Engine Triggers Classification

The workflow engine supports three primary classes of event triggers, executing either standard informational passes or advanced anomaly checking nodes.

Trigger Class Source Engine Action / Behavior Trigger Guardrails
Reactive Employee Chat UI Executes supervisor triage, selects Policy (RAG query + Cache) or Action (Tool parse) node, fanning out into parallel anomaly checks. Input check (PII/Injection), Output groundedness & compliance veto.
Scheduled System Cron Daemon Initiates periodic background checks (IQR, Z-Score) across all active datasets. Directly triggers anomaly node execution. Bypasses normal chat RAG; directly outputs summary narrative logs.
System-Generated Compliance Monitor Triggered internally when a transaction fails compliance checks (e.g. salary change exceeds allowed threshold). Vetoes action immediately. Bypasses parallel check routing; escalates to HITL queue.

Advanced Mode Guardrails & Vetoes

When operating in Advanced Mode, the system executes parallel checks on Action nodes. If an anomaly is severe (severity ≥ 0.8) or a hard compliance veto is triggered, the engine intercepts the execution path:

  • Interception: The parallel checks flag hitl_needed = True inside SharedState.
  • Escalation: Standard completion path is halted. A HITLRequest entry is saved with context containing the initial query, anomaly severity reports, and compliance veto reasons.
  • Manual Review: The query is displayed on the Human-in-the-Loop review dashboard (AGUI) until an administrator provides feedback.
  • Self-Learning Reinforcement: When the admin submits their verdict (approve or reject), the bandit model is rewarded or penalized using a Contextual Bandit update.

    • Deployment Guide

    Prerequisites: Python 3.10+, Node.js 18+, Docker (optional). Ensure you have your NVIDIA API keys ready.

    1. Environment Setup & Secret Keys

    Create a .env file in the project root containing your API configurations:

    # NVIDIA API for Embeddings and Triage/RAG LLM Router
NVIDIA_API_KEY=nvapi-your-key-here

# Mode Configuration (standard / advanced)
APP_ROLE=admin
STARTUP_REINDEX=true

# Tracing & Telemetry (Optional)
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.comenv
    2. Local Deployment (Backend & Frontend)

    Backend Server (FastAPI):

    # Initialize python venv
python -m venv venv
venv\Scripts\activate

# Install requirements
pip install -r backend/requirements.txt

# Start backend server on port 8000
python backend/main.pybash

    Frontend Portal (Vite + React):

    # Navigate to frontend folder
cd frontend

# Install package dependencies
npm install

# Run Vite dev server on port 5173
npm run devbash
    3. Docker Container Deployment

    Build and run backend and frontend nodes concurrently using docker-compose:

    version: '3.8'
services:
  backend:
    build:
      context: ./backend
    ports:
      - "8000:8000"
    environment:
      - NVIDIA_API_KEY=nvapi-your-key-here
      - STARTUP_REINDEX=true
    volumes:
      - ./backend/data:/app/backend/data

  frontend:
    build:
      context: ./frontend
    ports:
      - "80:80"
    depends_on:
      - backenddocker-compose.yml
    # Run composition in detached mode
docker-compose up -d --buildbash
    4. Enterprise Cloud Production Release (AWS / GCP)

    AWS ECS (Fargate) / EKS Strategy:

  • Push Docker images to AWS Elastic Container Registry (ECR).
  • Deploy backend service on AWS ECS Fargate task definition with CPU ≥ 2 vCPU and Memory ≥ 4 GB (critical for index indexing).
  • Configure AWS Application Load Balancer (ALB) to handle CORS preflight routing headers between frontend domain and backend target groups.
  • Mount an AWS EFS volume to /app/backend/data to persist ChromaDB index databases across container restarts.

    • Google Cloud Run Deploy:

    Observability & Tracing Setup

    Complete tracing telemetry maps every node action, cache comparison check, LLM response context, and tool output back to a centralized log.

    Trace Logs Preservation: Node trace records are preserved sequentially inside state.trace_log lists rather than overwritten, allowing trace viewers to display the full lifecycle of every execution step.

    Langfuse Instrumentation

    To review exact prompt tokens, latency distributions, and model costs in real time:

  • Configure Langfuse credentials inside the backend .env file.
  • Start the FastAPI server. Traces will automatically sync asynchronously via background task handlers without blocking user request threads.
  • Open the Langfuse dashboard to view trace hierarchies grouped by conversation session.
